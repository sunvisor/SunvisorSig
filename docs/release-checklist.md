# v1 Release Checklist

このプロジェクトは機能面ではかなり揃っているが、そのまま本番投入する前に確認すべき項目が残っている。

## 現在の到達点

- フォーラム / チャンネル / 投稿 / コメント
- 添付ファイルの追加 / 削除
- 招待 / アクティベーション / ログイン
- 権限制御
- 投稿ステータス
- アプリ内リアルタイム通知
- フォーラム単位の webhook 通知
- 監査ログ
- unit test / integration test 基盤

## 本番前に必須

### 環境変数

- `AUTH_SECRET` を十分に長いランダム文字列へ変更する
- `APP_URL` を本番 URL に変更する
- 招待メールを使う場合は `EMAIL_API_URL`, `EMAIL_API_TOKEN`, `EMAIL_FROM` を設定する
- webhook を使う場合は、各フォーラムごとに正しい webhook URL を登録する

### 初期データ

- 開発用 seed アカウントを本番に入れない
- `password123` 前提の seed データを本番 DB に投入しない
- 本番用の全体管理者アカウントを別途作成する

### セキュリティ

- `.env` を本番配布物に含めない
- webhook URL を secret として扱う
- 本番ログに機密情報を出さない
- 招待 URL を運用上どの経路で渡すかを決める

## 本番前に強く推奨

### 運用

- `npm run db:purge` を 1 日 1 回動かす cron を用意する
- 招待メール失敗時の運用手順を決める
- webhook 送信失敗時の監視方法を決める
- 管理者向けの運用手順書を用意する

### インフラ

- SSE は単一プロセス前提なので、複数インスタンス運用する場合は Redis pub/sub などへ置き換える
- 添付ファイルの保存先は現状ローカルファイルなので、冗長化やバックアップ方針を決める
- PostgreSQL のバックアップ方針を決める

### UX

- エラーメッセージ文言の最終確認
- webhook 設定時の入力補助文言の追加
- 削除済みデータ復元 UI が必要か判断する

## 後でよい

- パスワード再設定
- 利用者向け webhook 設定依頼フロー
- 削除済み投稿 / コメントの復元 UI
- E2E テスト
- webhook 配送の job queue 化

## リリース判定

機能面では v1 として成立している。  
ただし、本番投入前には最低でも以下が完了している必要がある。

- 本番用環境変数の設定
- seed データの非投入確認
- purge バッチの運用設定
- webhook / メール API の運用手順確認
- バックアップ方針の決定
